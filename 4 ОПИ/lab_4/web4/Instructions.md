# Спецификация и Инструкция по мониторингу и нагрузочному тестированию

Данный документ описывает архитектуру системы наблюдаемости (Observability), профилирования и нагрузочного тестирования лабораторной работы №4 по дисциплине «Основы программной инженерии».

---

## 1. Архитектурная схема взаимодействия компонентов

Сборка и работа системы построена на базе гибридной инфраструктуры, объединяющей физический хост Windows и виртуальную среду Linux (WSL 2):

```text
[ Windows Host ]                                     [ WSL 2 (Ubuntu) / Docker ]
┌──────────────────────────────┐                      ┌──────────────────────────────┐
│  IntelliJ IDEA               │                      │  Docker-compose:             │
│  └─ Spring Boot App (8080)   │ <─── (Scrape 5s) ─── │  └─ Prometheus (9090)        │
│  └─ HikariCP (Connection)    │                      │  └─ Grafana (3000)           │
│                              │ ◄─── (Load Test) ─── │  └─ Yandex Tank (Phantom)    │
│                              │                      │  └─ PostgreSQL (5432)        │
├──────────────────────────────┤                      └──────────────────────────────┘
│  Python Data Prep Script     │
│  └─ generate_ammo.py         │ ─── (Generates Binary) ──►  ammo.txt (Mapped Volume)
└──────────────────────────────┘
```

## 2. Сценарий демонстрации лабораторной работы (Шаг за шагом)

### Шаг 1. Запуск инфраструктуры мониторинга
В терминале **WSL** перейдите в корень проекта и запустите контейнеры базы данных, Prometheus и Grafana:
```bash
docker-compose up -d
```

---

### Шаг 2. Запуск приложения в режиме профилирования
1. Откройте проект в **IntelliJ IDEA**.
2. Запустите приложение через кнопку **"Run 'Web4Application' with Profiler"** (используя встроенный *async-profiler / Java Flight Recorder*).
3. Убедитесь, что приложение успешно запустилось на порту `8080` и в консоли `netstat` привязано к адресу `0.0.0.0:8080` (слушает все интерфейсы, включая мост WSL).

---

### Шаг 3. Динамическая генерация JWT-патронов
Поскольку приложение защищено JWT-авторизацией, перед обстрелом нужно получить свежие токены доступа.

1. Откройте **PowerShell / CMD в Windows** в папке `web4/tank`.
2. Запустите скрипт автоматической авторизации:
   ```powershell
   python get_token.py
   ```
---

### Шаг 4. Запуск нагрузочного тестирования (Yandex Tank)
В терминале **WSL** перейдите в папку `tank` и запустите обстрел:
```bash
WINDOWS_IP=$(ip route show | grep -i default | awk '{ print $3}')
docker run --rm \
    -v "$PWD:/var/loadtest" \
    --add-host host.docker.internal:$WINDOWS_IP \
    -it yandex/yandex-tank
```

---

### Шаг 5. Демонстрация результатов в Grafana
Откройте браузер на странице `http://localhost:3000` и покажите графики во время обстрела:

1. **Панель RPS (Requests Per Second):** Покажите, как график растет ступенями (10 -> 20 -> 30 -> 40 -> 50 RPS) в соответствии с файлом конфигурации `load.yaml`.
2. **Панель CPU Load (JVM):** Покажите, что нагрузка на процессор со стороны Java-процесса минимальна (около 5-7%).
3. **Панель JVM Heap Memory:** Покажите **пилообразный график** использования кучи (Heap).
4. **Панель бизнес-метрик:** Покажите рост графиков общего количества точек и доли попаданий.

## Какие метрики есть

### Базовые метрики

- **CPU usage**
  - Сейчас на дашборде используется метрика JVM/process:
    - `process_cpu_usage`
  - Она показывает загрузку CPU именно текущим Java-процессом приложения.

- **Memory usage**
  - Сейчас на дашборде используется метрика JVM heap:
    - `jvm_memory_used_bytes{area="heap"}`
    - `jvm_memory_max_bytes{area="heap"}`
  - Отображается процент занятый heap-памяти JVM.

- **RPS total**
  - `http_server_requests_seconds_count`
  - Показывает общий поток HTTP-запросов.

- **RPS mean**
  - Вычисляется через rate/count по `http_server_requests_seconds_count`
  - Показывает средний RPS по маршрутам.

### Продуктовые метрики

- **Количество точек в области**
  - `app_hits_inside_total`
  - Сколько точек попало в область.

- **Общее количество точек**
  - `app_hits_total`
  - Сколько всего точек отправил пользователь.

- **Точки по пользователю**
  - `app_user_points_total{user,username}`
  - Счётчик точек для каждого пользователя.

- **Средний интервал между кликами**
  - `app_click_interval_ms{user,username}`
  - Показывает среднее время между действиями пользователя.

- **Средняя задержка обработки addHit**
  - `app_add_hit_duration_ns`
  - Показывает, сколько времени занимает проверка точки.
